## 投影片 1 ── 封面（約 35 秒）

各位老師、各位同學大家好，我們這組要報告的專案是《Bullet Storm》，一款用純 Java 從零打造的彈幕射擊遊戲。

這份簡報專注在「圖形介面」這個面向——也就是玩家實際看到、操作到的一切。整個遊戲是一個 800×600 的視窗，左邊 480×560 是遊戲戰場，右邊是一塊資訊面板。我們提供兩位可選角色、四種難度。

接下來我會依照五個段落來講：先講專案動機，再講系統架構，然後是核心技術，接著用實驗數據驗證，最後談未來改進。

> （過場）先從「我們為什麼要做這個介面」開始。

## 投影片 2 ── 專案動機 Project Motivation（約 60 秒）

我們選擇這個題目，背後有四個動機。

第一，**遊戲介面的技術挑戰**。彈幕遊戲必須在六十分之一秒內，把數百顆子彈加上整塊 HUD 全部畫完。光是好看不夠，必須同時兼顧效能，這是很硬的工程考驗。

第二，**純 Java、零依賴**。我們刻意不用任何遊戲引擎或 UI 框架，只靠 Java Swing 跟 Graphics2D，從零打造完整的視覺系統，藉此證明我們對底層渲染的掌握。

第三，**資訊視覺化設計**。在高密度的彈幕戰場裡，玩家要同時判讀分數、生命、等級——色彩、佈局、動畫缺一不可。

第四，**無障礙與可玩性**。我們讓對比度符合 WCAG AA、色彩一定搭配圖示、全鍵盤就能操作，確保不同玩家都玩得順。

一句話總結我們的目標：介面是玩法的延伸，不是事後貼上的裝飾。

> （過場）有了動機，先看整個畫面是怎麼切分的。

## 投影片 3 ── 系統架構：視窗佈局與座標（約 75 秒）

這張圖是整個視窗的配置，左邊是示意圖、右邊是對應的技術規格。

最外層青色框是 800×600 的總視窗，固定大小、置中，我們用 Insets 去計算內部真正可用的區域。

中間綠色框是真正的遊戲區，從座標 (32, 16) 開始，大小 480×560。這些數值不是隨手寫死的，而是定義成 `FIELD_X`、`FIELD_Y`、`FIELD_W`、`FIELD_H` 常數，全程式都引用同一組，要調整時改一處就好。這塊區域用「雙重緩衝」渲染——先把整張畫面畫進一張離螢幕的 BufferedImage，再一次貼到螢幕，這樣六十 FPS 下完全不閃爍、不撕裂。背景還有 150 顆會動的星星營造太空感。

右邊金色框是 HUD 面板，它的 x 座標是動態算出來的：`HUD_X = FIELD_X + FIELD_W + 16`，剛好等於 544，裡面依序放分數、炸彈、生命、等級、EXP 條、五項技能。

最後，整個迴圈的心臟是 `javax.swing.Timer`，每 16 毫秒觸發一次，驅動更新跟重繪。

## 投影片 4 ── UI 狀態機與畫面流程（約 60 秒）

整個遊戲用一個「狀態機」管理，總共六個狀態，由 `GameState` 這個列舉來控制。

玩家一進來是 **START 開始畫面**，有標題動畫跟星場背景；接著是 **角色選擇**，兩張角色卡片做對比；然後 **難度選擇**，四級難度用顏色區分；確定後進入 **PLAYING 遊戲進行**，HUD 即時更新。中途按暫停會進 **PAUSED**，蓋上半透明遮罩跟選單；角色陣亡或通關則進 **GAME OVER**，顯示統計面板跟重試。

設計重點是：每個狀態都有自己獨立的 `render()` 跟 `handleInput()`，彼此不干擾；狀態之間切換還加了大約 0.3 秒的淡入淡出，讓過場不會太生硬。

> （過場）了解流程後，來看支撐這一切的核心渲染技術。

## 投影片 5 ── 核心渲染技術 Technologies Used（約 80 秒）

這頁是六項核心技術。

第一，**Java Swing 跟 JPanel**：我們繼承 JPanel、覆寫 `paintComponent()`，再用 KeyListener 接輸入，靠 Swing 的事件驅動視窗系統跨平台運作。

第二，**雙重緩衝**：剛剛提過，先畫進離螢幕影像再一次貼出，徹底消除閃爍。

第三，**Graphics2D API**：它提供抗鋸齒、半透明合成、座標旋轉位移這些向量繪圖能力，是我們所有特效的基礎。

第四，**Timer 主迴圈**：每 16 毫秒跑一次「更新再繪製」，穩定維持六十 FPS。

第五，也是我特別想強調的——**色彩編碼系統**。我們讓顏色本身帶有意義：紅色是危險、綠色是收益、藍色是一般資訊、金色保留給最重要的分數。玩家不用讀文字，光看顏色就能反應。

第六，**粒子系統**：擊殺敵人時噴出爆炸粒子，每顆有獨立的速度、壽命跟縮放，全部用純 Java 2D 畫出來，沒有任何外部函式庫。

> （過場）技術講完，來看玩家戰鬥時最常盯著的右側面板，跟角色、難度設計。

## 投影片 6 ── HUD 設計、角色與難度系統（約 80 秒）

這是整份簡報的重點之一，左右兩塊。

**左邊是 HUD 面板的八個元素**。最上面是分數，用大字金色，因為它是玩家最在意的成就感；接著是炸彈，圖示加數字；生命用綠色條形顯示剩餘量；再來是這款遊戲的特色——等級跟紫色的 EXP 升級條；最下面是五項技能的星級顯示，從零顆到五顆星，分別是射速、射頻、生命、炸彈、刷掠。

**右邊是難度系統**，四級多維度調整。請看這張表：難度不是只改一個數字，而是同時動三個軸——子彈速度、每波數量、敵人 AI 積極度。Easy 是基準的 0.65 倍、每波六到八發、AI 緩慢；一路到 Lunatic 拉到 1.04 倍、十到十二發、全速進攻。底下的速度公式是 `speedMult =（1.0 + 難度×0.2）× 0.65`，所以四級分別是 0.65、0.78、0.91、1.04 倍，曲線平滑遞增，不會突然跳難。

> （過場）設計講完，到底跑起來表現如何？用數據說話。

## 投影片 7 ── 實驗結果：渲染效能與可用性（約 65 秒）

左邊是四項效能指標。第一，**幀率穩定度**：六十 FPS，誤差只有正負百分之一。第二，**HUD 渲染成本**：每幀重繪低於 0.2 毫秒，因為我們只重畫變動的部分。第三，**文字對比度**：全部達到 WCAG AA 的 4.5:1 標準。第四，**記憶體占用**：靠物件複用壓在 80 MB 以下。

右上角這條折線是六十秒實測的 FPS 曲線，幾乎是一條貼著 60 的平線，證明長時間遊玩也不掉幀。

右下角是可用性測試摘要：新玩家三十秒內就看懂 HUD 佈局；色彩編碼降低了資訊誤判；全鍵盤操作不需要滑鼠；深色主題長時間玩也護眼。

這些數據說明——好的介面，是「好看」跟「好用」同時成立。

> （過場）目前成果不錯，但我們還有很多想做的。

## 投影片 8 ── 未來改進方向 Future Improvements（約 55 秒）

我們把改進分成短、中、長三期。

**短期**，著重在打磨現有體驗：加入設定選單，可調音量、解析度、自訂按鍵；強化暫停選單；還有 HUD 主題一鍵切色盤。

**中期**，擴充輸入跟內容：支援遊戲手把、加入成就系統做結算視覺化、增加更多角色皮膚。

**長期**，是比較大的方向：做行動版的觸控 UI 跟虛擬搖桿、線上排行榜的即時比分介面，甚至讓玩家自己拖曳排版、自訂 HUD 佈局。

可以看到，這些改進全都圍繞同一個核心——讓介面更貼近玩家。

> （過場）做個總結。

## 投影片 9 ── 結語（約 35 秒）

總結圖形介面這塊：800×600 的最佳化視窗、480×560 的戰場、八種 HUD 元素、四級平滑遞增的難度，所有設計都圍繞一個核心——讓玩家「看得懂、反應得來、玩得舒服」。

介面不是事後才貼上去的皮，而是從一開始就和玩法一起設計的。原始碼都放在 GitHub 上，歡迎參考。以上就是圖形介面的部分，謝謝大家，有任何問題歡迎提問。

### 問答備用（可能被問到的）

- **Q：為什麼選 Java Swing 而不是現成遊戲引擎？**
  A：為了展示我們對底層渲染與遊戲迴圈的掌握，不依賴現成引擎，所有東西都自己實作；這也是專案的學習目標之一。

- **Q：六十 FPS 下這麼多子彈，HUD 不停更新會不會卡？**
  A：不會。HUD 每幀只重畫變動的部分，實測低於 0.2 毫秒；整體碰撞偵測也在 1 毫秒內，細節在演算法篇。

- **Q：雙重緩衝跟直接畫差在哪？**
  A：直接畫會在繪製過程中被螢幕讀到「半成品」，造成閃爍跟撕裂；雙重緩衝先在記憶體畫完整張，再一次貼出，畫面永遠是完整的。

- **Q：色彩編碼對色盲玩家友善嗎？**
  A：友善。我們的原則是「顏色一定搭配圖示或文字」，紅色危險一定配圖示，不單靠顏色傳達資訊，符合無障礙設計。
